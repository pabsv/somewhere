## Architectural Decisions

- **Price-band colouring, not discount tiers (2026-07-20):** the whole product reframed from "good discounts" ‚Üí "very cheap flights, absolute". The `deal_tier` (`steal`/`deal`/`fair`) enum now derives from **absolute round-trip price**, not from score-vs-baseline. Single change point: `frontend/lib/score.ts` ‚Äî new `PRICE_BAND_STEAL=50` / `PRICE_BAND_DEAL=100` + `tierForPrice(price)` (‚â§‚Ç¨50 steal / ‚â§‚Ç¨100 deal / else fair); `scoreTrip` sets `deal_tier = tierForPrice(price)`. Because **every** colour map keys off `deal_tier` (untouched `TIER_BAR`/`TIER_CLASSES`/`TIER_BADGE`), all bars/chips/badges recolour for free. `score`/`delta_pct` and the whole baseline plumbing (`getBaselines`, `scrape_targets.price_p50_ewma`, `toTrip` lookup, `MAX_DEAL_PRICE` score-cap) are **kept but dormant** ‚Äî revert = restore the 6-line tier derivation; no schema/API/Python change. `promoteFavouriteTier` redefined on relaxed bands (`FAV_PRICE_BAND_STEAL=75`/`FAV_PRICE_BAND_DEAL=130`, signature kept as `(tier,_score,price)` so the 4 call sites are untouched); old `FAV_*_SCORE_THRESHOLD` dropped; old `STEAL/DEAL_SCORE_THRESHOLD` kept exported but marked DORMANT (tests import them). **Copy:** all relative "% below typical" (`belowPct`) removed from `CityCard`/`TripRow`/`TripTooltip`/`TripPopover` (the actual discount framing); STEAL/DEAL badge *words* kept (read fine as very-cheap/cheap); open-jaw "‚Ç¨X under round trip" / "mix & match" / twin-city hints kept (split-ticket savings, not baseline-discount). **Ranking ‚Üí cheapest-first everywhere:** `getCitiesData` best-pick + city sort and `getCityData` trip sort switched from score-desc to price-asc (calendar `curateBars`/`assignLanes` + Explore client grid were already price-first). Open-jaw combos on the calendar (`toCalTrip`) now colour by `tierForPrice(oj.total_price)` instead of `vs_roundtrip`, for board consistency. `isNearAvailWorthy` unchanged (now effectively price-only since steal‚ü∫‚â§‚Ç¨50). Verified: `tsc`+`next build` green; `score.test.ts` (band boundaries + `tierForPrice`) + `near-avail.test.ts` (8) pass; live on :4176 ‚Äî Explore FareTags compute exact tokens (‚Ç¨38‚Üí`rgb(14,159,110)` steal-green, ‚Ç¨51‚Üí`rgb(255,199,44)` deal-yellow, ‚Ç¨101‚Üíwhite fair), no "% below typical" copy, grid cheapest-first (‚Ç¨38‚Üí‚Ç¨40‚Üí‚Ä¶); `/api/trips` 254 trips **0 tier/price mismatches** (123 steal / 77 deal / 54 fair). Calendar bars not eyeballed live (browser pane went 0x0 ‚Äî documented flake ‚Äî and signed-out `avail=true` yields 0 bars) but the data path is proven identical. Not committed by this session. Tuning = the two band constants. Follow-up ideas (not built): a `?fav=` param so favourite-promoted bars survive the server-side `?tier=` filter; a colour legend.

- **Calendar `/api/trips` data-cache (2026-07-20):** the calendar crawled in **local dev** (13‚Äì71s, sometimes looked like it never loaded) while prod was fast. Root cause = two things, neither a code bug: (1) up to **5 concurrent dev servers** running (`4173‚Äì4177`) ‚Üí Turbopack CPU starvation turned a ~5s job into 70s (measured: one `/api/trips` warm hit hit 71s with 5 servers up; ~2.4s with 1); (2) `/api/trips` had **zero caching** (`force-dynamic`, no `unstable_cache`) so every load/refresh re-ran two full Atlas `.toArray()` reads (density + bars) + scoring ‚Äî and from a home/uni link each round-trip is slow (raw measured: 3s cold connect, 3.4s bars fetch of 3386 full docs ‚âà 2.5MB, IXSCAN so not a missing index; prod is fast only because Vercel‚ÜîAtlas is a same-datacenter link). Fix (`lib/queries.ts`): split `getTripsData` into **user-independent halves that are `unstable_cache`d 2 min** (`TRIPS_REVALIDATE_SECONDS`, matching `/api/cities`) ‚Äî `cachedDensitySource(origins,start,end)` (compact `DensitySourceRow[]`, dep/arr hours precomputed) and `cachedScoredBars({origins,start,end,maxPrice,direct,minNights,maxNights})` (`dedupeTrips(scoreFlights(...))` = `Trip[]`, calls `getBaselines` inside the cache exactly like cities does). The **avail / near-miss / tier / curate** passes stay per-request in-memory over the cached halves, so a signed-in "Only my free dates" reload (the default ‚Äî `onlyFree=true`) also skips the Atlas re-read (only the small `loadUserAvailability` query + cheap in-memory `passesAvail` filter rerun) and **availability edits are never served stale** (filtering isn't cached). Cache keys sort origins; both cached values are plain JSON (no in-place mutation downstream ‚Äî near-miss spreads, curate builds new arrays ‚Äî so the shared cached refs are safe). Result: cache HIT ~2.4s in dev (was 13‚Äì71s), MISS ~7s incl. route compile; prod effectively free on hits. tsc green, live 200 with identical payload shape verified on :4177. **Operational note:** don't run more than ~1‚Äì2 dev servers at once ‚Äî the CPU contention dominates everything else (this is also why the CSS-500 cache-corruption bug worsens with 3+ servers, logged below). Not committed by this session.

- **Ground-competitive destination cap (2026-07-19):** destinations close enough to the NL/BE origins that a train/Flixbus is a real alternative only surface on the discovery boards when the whole round trip is ‚â§ `GROUND_COMPETITIVE_MAX_PRICE` (‚Ç¨100) ‚Äî above it the ground option wins and the flight was just clutter (user reported Paris / near-German listings as "stupid"). A genuinely cheap one (e.g. a sub-‚Ç¨100 Berlin steal) still shows, per the user's "take it if it's a steal" rule. Pieces in `lib/score.ts`: `GROUND_COMPETITIVE_CODES` (set ‚Äî **all Germany** BER/MUC/FRA/HAM/STR/HAJ/NUE/LEJ/DRS/BRE/FMM/FKB/FMO + Luxembourg LUX + **northern France** CDG/ORY/BVA/LIL; southern France / Switzerland / UK deliberately uncapped ‚Äî real flights), `GROUND_COMPETITIVE_MAX_PRICE`, `isGroundCompetitive()`. Enforced in `lib/queries.ts` via one `GROUND_COMPETITIVE_NOR` Mongo `$nor` clause ("NOT (ground-competitive AND >cap)", composes without touching the existing `price` key) merged into **`buildTripFilter`** (Calendar bars+density, DaySheet, Group calendar) and the **`getCitiesData`** match (Explore grid, FavouritesStrip, landing DepartureBoard). **City drill-in (`getCityData`) is EXEMPT** ‚Äî a deliberate `/city/[code]` visit / SearchCombobox pick shows the full picture; the cap only de-clutters the boards. So a capped-out city (no sub-‚Ç¨100 fare) vanishes from Explore/Calendar but stays reachable by search. Verified live 2026-07-19: 170/170 current ground-competitive future flights are >‚Ç¨100 (cheapest BER ‚Ç¨120), so all are currently hidden. Tuning = one constant / edit the set. Follow-ups not built: (1) let a **favourited** city bypass the cap (user explicitly wants it) ‚Äî needs per-user set in the query; (2) apply to open-jaw/twin combos (currently exempt ‚Äî niche + already curated).
