## Architecture

```
Vercel (Next.js frontend)
    ↕  Next.js API routes
MongoDB Atlas (flight_scraper database)
    ↕  pymongo
Home PC (Python scheduler + scraper)
```

Frontend and backend never talk directly — both connect to Atlas. Same pattern as Boulevard Connection.

## How to Run

```bash
# ─── Frontend ────────────────────────────────────────────────────────────
# Deployed on Vercel (production). Local dev:
cd frontend && npm run dev        # http://localhost:4173

# ─── Pool scheduler (NEW — production path) ──────────────────────────────
# Pool-based, user-agnostic. Pulls due routes from `scrape_targets`.
indexes           # one-time: create indexes + TTLs on Atlas
seed              # one-time: upsert 1150 routes into scrape_targets (staggered 24h)
seed --stats      # show pool stats
pool              # continuous loop, 24h (00:00–24:00), 2-min slots
pool --once       # claim & scrape one due route, exit
pool --route EIN BCN  # force-run a specific route, exit
pool --dry-run    # list next 20 due routes, scrape nothing

# Full commands behind the shortcuts:
python -m database.setup_indexes
python -m scripts.seed_targets [--stagger MIN] [--prune | --prune-delete] [--stats]
python -m scheduler.pool_scheduler [--once | --route O D | --dry-run | --direct-only]

# ─── Legacy user-driven scheduler (still works, but not the prod path) ───
simulate          # 60 min/cycle dev mode
scrape            # run all origins once and exit
scrape EIN        # run single origin once

# ─── Misc ────────────────────────────────────────────────────────────────
api               # FastAPI on port 9000 (rarely needed)
uvicorn api.main:app --reload --port 9000
```

> Shortcuts are `.bat` files in the project root — work in cmd, PowerShell, and Git Bash.

## Architectural Decisions

- **Reach-scaled price bands (2026-07-21):** the flat `€50/€100` bands treated €56 to Tangier and €56 to Bergamo as the same fare, and they are not — one is 2 100 km to another continent, the other is a 1 h hop. Bands are now multiplied by a **destination reach multiplier**. Single change point, still `frontend/lib/score.ts`: new `REGION_REACH_MULTIPLIER` (keyed on the `region` strings from `scraper/targets.py` → `data/destinations.gen.ts`), `DEFAULT_REACH=1.0`, `reachMultiplier(dest?)`, `bandsForDest(dest?)`; `tierForPrice(price, dest?)` / `scoreTrip(price, baseline, dest?)` / `promoteFavouriteTier(..., dest?)` / `isNearAvailWorthy(..., dest?)` all take an **optional** IATA code, so any caller that omits it behaves exactly as before (unknown codes also fall back to 1.0 — an un-exported destination can never get a looser band by accident). Table (great-circle from EIN, hand-rounded): core Europe (Germany/Alps, France, UK & Ireland, Central Europe, Italy) 1.0 · Iberia/Balkans/Nordics 1.1 · SE Europe 1.15 · Greece 1.3 · **N. Africa + Türkiye 1.5** · Levant 1.9 · Caucasus 2.0 · Atlantic 2.3 · Gulf 2.6. So N. Africa steals ≤€75, deals ≤€150. **Deliberately NOT price-relative:** the multiplier is a reach/distance measure, not the region's observed p50 — scaling by observed price would silently reintroduce the discount framing killed on 07-20 ("cheap for Istanbul" ≠ "cheap"). `MAX_DEAL_PRICE=400` and `HARD_PRICE_CEILING` stay absolute and still win. Call sites threaded: `lib/trips.ts` (`toTrip` → `scoreTrip(..., doc.destination)` — covers every board via `scoreFlights`), `app/calendar/page.tsx` (`toCalTrip` combos, keyed on the fly-in city; favourite promotion), `CityDetail`, `CityCard`, `FavouritesStrip`, `lib/queries.ts` near-miss gate. **Measured impact on live Atlas** (5 428 future fares ≤€400): 302 recolour (5.6%) — steal 412→466, deal 1 062→1 310, fair 3 954→3 652; biggest movers Iberia (84 fair→deal, 22 deal→steal), N. Africa (90/15), Greece 42, Türkiye 41. No flood. Verified: `tsc` + `next build` green, `score.test.ts` (extended: multipliers, band scaling, the €56 TNG-vs-BGY case, MAX_DEAL_PRICE precedence, favourite/near-miss scaling) + `near-avail.test.ts` (8) pass; live on :4173 — `/api/cities/TNG` returns €56 `steal` (was `deal`), `/city/TNG` FareTags compute `rgb(14,159,110)` steal-green, AGA €103 / RAK €106 now `deal` (were `fair`), BGY €36 unchanged `steal`, 0 failed requests. Not committed by this session. Tuning = the one table. **Known limit:** this is the "we only fly this pool" approximation — region granularity means Egypt (3 300 km) and Morocco (2 100 km) share 1.5. Replace the table with real per-airport great-circle distance once the pool leaves this region set (needs lat/lon, which `targets.py` does not carry today). Also unscaled: `CALENDAR_DEFAULT_MAX_PRICE=200` and `GROUND_COMPETITIVE_MAX_PRICE=100` (the latter correctly — those regions are all reach 1.0).

- **No durable price history (2026-07-21, audit):** answering "can we analyse real price trends later?" — **not today, nothing stores a time series.** `price_history` has a model (`database/models/price_history.py`), a repo (`database/repositories/price_history_repo.py`), a config constant and a 180d TTL index, but **zero writers** — `scripts/migrate_v2.py` dropped the collection in the v2 migration and nothing re-created it; it is absent from Atlas (`list_collection_names()` has no `price_history`, no `route_stats`). What does exist is all short-lived or lossy: `flights.price_points[] {p, at}` is a real per-flight price log (sample `EIN-LHR-2026-08-21-2026-08-24` carries 5 points) but dies with the **14d `last_seen_at` TTL**; `oneway_fares.prices` is **replaced wholesale** each scrape (21d TTL) so it has no history at all; `scrape_targets.price_p50_ewma` is a single rolling number, not a series; `scrape_runs` (30d) logs runs, not fares. Consequence: any "is this route trending down / is September always cheap" question is unanswerable beyond a 14-day window. Fix when wanted = a small append-only rollup written by the pool scheduler (per route per scrape: min/p50/count + a date-bucket key), no TTL or a long one — deliberately NOT built in this session per the user's scope call.

- **Price-band colouring, not discount tiers (2026-07-20):** the whole product reframed from "good discounts" → "very cheap flights, absolute". The `deal_tier` (`steal`/`deal`/`fair`) enum now derives from **absolute round-trip price**, not from score-vs-baseline. Single change point: `frontend/lib/score.ts` — new `PRICE_BAND_STEAL=50` / `PRICE_BAND_DEAL=100` + `tierForPrice(price)` (≤€50 steal / ≤€100 deal / else fair); `scoreTrip` sets `deal_tier = tierForPrice(price)`. Because **every** colour map keys off `deal_tier` (untouched `TIER_BAR`/`TIER_CLASSES`/`TIER_BADGE`), all bars/chips/badges recolour for free. `score`/`delta_pct` and the whole baseline plumbing (`getBaselines`, `scrape_targets.price_p50_ewma`, `toTrip` lookup, `MAX_DEAL_PRICE` score-cap) are **kept but dormant** — revert = restore the 6-line tier derivation; no schema/API/Python change. `promoteFavouriteTier` redefined on relaxed bands (`FAV_PRICE_BAND_STEAL=75`/`FAV_PRICE_BAND_DEAL=130`, signature kept as `(tier,_score,price)` so the 4 call sites are untouched); old `FAV_*_SCORE_THRESHOLD` dropped; old `STEAL/DEAL_SCORE_THRESHOLD` kept exported but marked DORMANT (tests import them). **Copy:** all relative "% below typical" (`belowPct`) removed from `CityCard`/`TripRow`/`TripTooltip`/`TripPopover` (the actual discount framing); STEAL/DEAL badge *words* kept (read fine as very-cheap/cheap); open-jaw "€X under round trip" / "mix & match" / twin-city hints kept (split-ticket savings, not baseline-discount). **Ranking → cheapest-first everywhere:** `getCitiesData` best-pick + city sort and `getCityData` trip sort switched from score-desc to price-asc (calendar `curateBars`/`assignLanes` + Explore client grid were already price-first). Open-jaw combos on the calendar (`toCalTrip`) now colour by `tierForPrice(oj.total_price)` instead of `vs_roundtrip`, for board consistency. `isNearAvailWorthy` unchanged (now effectively price-only since steal⟺≤€50). Verified: `tsc`+`next build` green; `score.test.ts` (band boundaries + `tierForPrice`) + `near-avail.test.ts` (8) pass; live on :4176 — Explore FareTags compute exact tokens (€38→`rgb(14,159,110)` steal-green, €51→`rgb(255,199,44)` deal-yellow, €101→white fair), no "% below typical" copy, grid cheapest-first (€38→€40→…); `/api/trips` 254 trips **0 tier/price mismatches** (123 steal / 77 deal / 54 fair). Calendar bars not eyeballed live (browser pane went 0x0 — documented flake — and signed-out `avail=true` yields 0 bars) but the data path is proven identical. Not committed by this session. Tuning = the two band constants. Follow-up ideas (not built): a `?fav=` param so favourite-promoted bars survive the server-side `?tier=` filter; a colour legend.

- **Calendar `/api/trips` data-cache (2026-07-20):** the calendar crawled in **local dev** (13–71s, sometimes looked like it never loaded) while prod was fast. Root cause = two things, neither a code bug: (1) up to **5 concurrent dev servers** running (`4173–4177`) → Turbopack CPU starvation turned a ~5s job into 70s (measured: one `/api/trips` warm hit hit 71s with 5 servers up; ~2.4s with 1); (2) `/api/trips` had **zero caching** (`force-dynamic`, no `unstable_cache`) so every load/refresh re-ran two full Atlas `.toArray()` reads (density + bars) + scoring — and from a home/uni link each round-trip is slow (raw measured: 3s cold connect, 3.4s bars fetch of 3386 full docs ≈ 2.5MB, IXSCAN so not a missing index; prod is fast only because Vercel↔Atlas is a same-datacenter link). Fix (`lib/queries.ts`): split `getTripsData` into **user-independent halves that are `unstable_cache`d 2 min** (`TRIPS_REVALIDATE_SECONDS`, matching `/api/cities`) — `cachedDensitySource(origins,start,end)` (compact `DensitySourceRow[]`, dep/arr hours precomputed) and `cachedScoredBars({origins,start,end,maxPrice,direct,minNights,maxNights})` (`dedupeTrips(scoreFlights(...))` = `Trip[]`, calls `getBaselines` inside the cache exactly like cities does). The **avail / near-miss / tier / curate** passes stay per-request in-memory over the cached halves, so a signed-in "Only my free dates" reload (the default — `onlyFree=true`) also skips the Atlas re-read (only the small `loadUserAvailability` query + cheap in-memory `passesAvail` filter rerun) and **availability edits are never served stale** (filtering isn't cached). Cache keys sort origins; both cached values are plain JSON (no in-place mutation downstream — near-miss spreads, curate builds new arrays — so the shared cached refs are safe). Result: cache HIT ~2.4s in dev (was 13–71s), MISS ~7s incl. route compile; prod effectively free on hits. tsc green, live 200 with identical payload shape verified on :4177. **Operational note:** don't run more than ~1–2 dev servers at once — the CPU contention dominates everything else (this is also why the CSS-500 cache-corruption bug worsens with 3+ servers, logged below). Not committed by this session.

- **Ground-competitive destination cap (2026-07-19):** destinations close enough to the NL/BE origins that a train/Flixbus is a real alternative only surface on the discovery boards when the whole round trip is ≤ `GROUND_COMPETITIVE_MAX_PRICE` (€100) — above it the ground option wins and the flight was just clutter (user reported Paris / near-German listings as "stupid"). A genuinely cheap one (e.g. a sub-€100 Berlin steal) still shows, per the user's "take it if it's a steal" rule. Pieces in `lib/score.ts`: `GROUND_COMPETITIVE_CODES` (set — **all Germany** BER/MUC/FRA/HAM/STR/HAJ/NUE/LEJ/DRS/BRE/FMM/FKB/FMO + Luxembourg LUX + **northern France** CDG/ORY/BVA/LIL; southern France / Switzerland / UK deliberately uncapped — real flights), `GROUND_COMPETITIVE_MAX_PRICE`, `isGroundCompetitive()`. Enforced in `lib/queries.ts` via one `GROUND_COMPETITIVE_NOR` Mongo `$nor` clause ("NOT (ground-competitive AND >cap)", composes without touching the existing `price` key) merged into **`buildTripFilter`** (Calendar bars+density, DaySheet, Group calendar) and the **`getCitiesData`** match (Explore grid, FavouritesStrip, landing DepartureBoard). **City drill-in (`getCityData`) is EXEMPT** — a deliberate `/city/[code]` visit / SearchCombobox pick shows the full picture; the cap only de-clutters the boards. So a capped-out city (no sub-€100 fare) vanishes from Explore/Calendar but stays reachable by search. Verified live 2026-07-19: 170/170 current ground-competitive future flights are >€100 (cheapest BER €120), so all are currently hidden. Tuning = one constant / edit the set. Follow-ups not built: (1) let a **favourited** city bypass the cap (user explicitly wants it) — needs per-user set in the query; (2) apply to open-jaw/twin combos (currently exempt — niche + already curated).

- **Pool scraping (2026-05-28):** decoupled scraping from users. Pool of 1150 (origin, dest) routes lives in `scrape_targets`, tier A/B/C cadenced (24h / 72h / 168h). Scheduler claims one route per 2-min slot, 24h (widened from 07-23 on 2026-07-14 — box runs always-on). Users will be matched to the pool later, but scraping does not depend on user prefs.
- **Origins (5):** EIN, AMS, BRU, CRL, MST — airports closest to Eindhoven with usable coverage. MST (Maastricht) added 2026-07-14 by user request; small airport, so most of its routes will come back empty and auto-disable. DUS / NRN commented out in `targets.py` for easy re-add.
- **Currency pinned to EUR (2026-07-14):** Google Flights picks the response currency itself (GeoIP) unless the request URL pins it — `?gl=NL&hl=en&curr=EUR` is appended to both fli endpoints in `FliScraper.__init__`. Before the pin, ~30% of stored flights were GBP/DKK/CZK/HKD/... with their raw numbers treated as EUR. A save-time guard in `flight_service.py` drops any non-EUR fare as defense in depth. Legacy non-EUR docs age out via the 14d flights TTL.
- **Destinations (230):** Europe + N.Africa + Levant + Türkiye + Atlantic. Curated in `scraper/targets.py`. Tier A=28 popular, B=84 mid, C=118 long-tail.
- **Per-route scrape recipe (2026-07-17 — any-duration one-way sweeps):** Phase 1 = 2 one-way SearchDates sweeps (O→D over 181d ≈ 3 HTTP calls after fli's internal 61-day chunking; D→O over ~189d ≈ 4 calls); every (out, ret) pair with 2–10 nights combined in memory (estimated price = sum of one-ways). Phase 2 = top 10 cheapest pairs (max 2 per departure date — `SCRAPE_MAX_PER_OUT_DATE`) via SearchFlights, real round-trip prices stored. ≈17 fli HTTP calls per route per cycle. If one-way grids come back empty, falls back to round-trip grids (durations 2/6/10) so the route doesn't count toward auto-disable. Knobs: `SCRAPE_MIN_NIGHTS` / `SCRAPE_MAX_NIGHTS` / `SCRAPE_TOP_N_CHEAP_DATES` in `database/config.py` (replaced `SCRAPE_DURATION_BUCKETS`).
- **One-way fare grids / open-jaw foundation (2026-07-17):** the Phase-1 one-way SearchDates grids (both directions, previously discarded after pair ranking) are now persisted to a new `oneway_fares` collection — **one doc per directed leg** (`leg_key` = `"EIN-BCN"`, `prices: {date: cheapest one-way €}` replaced wholesale each scrape, ~2300 docs ≈ 12MB, TTL 21d on `scraped_at`). Zero extra API calls. Pieces: `database/models/oneway_fare.py`, `database/repositories/oneway_fare_repo.py`, `FlightService.save_oneway_grids` (past-date + price-sanity filter; empty result skips the leg so the old grid survives), scraper exposes grids via `stats["oneway_grids"]`, pool scheduler saves best-effort/non-fatal (skipped on `--direct-only`). Purpose: future open-jaw/multi-city = two `find_by_leg` lookups + in-memory date join — origin-side (`EIN→BCN` × `BCN→AMS`) and destination-side (`EIN→BCN` × `MAD→EIN`) both covered since every pool route is swept both directions; combo price = sum of one-ways = real bookable price (two tickets). No UI/read path built yet. Caveat: return-direction grids start ~today+min_nights.
- **TTL cleanup:** `flights.last_seen_at` 14d (re-scraping refreshes the touch), `scrape_runs.started_at` 30d, `price_history.scraped_at` 180d, `oneway_fares.scraped_at` 21d. Past-date trips drop naturally because SearchDates only returns future dates.
- **Auto-disable:** a route is `enabled=false` after 5 consecutive empty/error runs. `seed --prune` resets orphans; `repo.reenable_all()` resurrects disabled routes.
- **Fli backend:** `scraper-fli/scraper.py`. Single instance per slot. No NordVPN rotation yet — add only if Google starts rate-limiting (DormSpot pattern).
- Azair only — Kiwi dropped (too rate-limited). Azair scraper kept as fallback backend.
- Multi-user, no password — identity via `X-User-ID` header (stored in localStorage). Login is name + email only.
- Frontend connects directly to Atlas via Next.js API routes (serverless) — no backend HTTP server needed
- **Landing / Explore split (2026-06-13):** `/` (`app/page.tsx`) is now a promo landing page — pitch headline ("Open to go anywhere, on any free day."), the live cheapest-steals `DepartureBoard`, CTA → `/explore`. `/explore` (`app/explore/page.tsx`) is the moved browsing page: board + grid. - **Interest cities / saved (2026-06-13):** users star destinations as focus points. Stored as `users.saved_cities: string[]` (uppercase IATA, sanitized + capped 500). Route `app/api/saved-cities/route.ts` (GET + PUT replace-all, session-gated, mirrors `/api/availability`). Stars render only when `signedIn`. Saved-only filter auto-clears if the last star is removed.
- **Favourites (upgrade of saved cities, 2026-07-17):** the ★ star IS now the "favourite" concept — a place the user *wants* (would take on a merely-good fare) or a "home" to fly to whenever cheap + free. **No storage change** — still `users.saved_cities` + `/api/saved-cities` + `useSavedCities()` (only header comment reworded; provider/hook/route names unchanged, no migration).
